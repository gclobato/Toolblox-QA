📖 Wiki QA – Quality Assurance

Fundamentos de QA

O que é QA: Garantia de qualidade no desenvolvimento de software, focando em prevenção de falhas e melhoria contínua.

QA x Tester:

  • QA → processos e métricas.
  • Tester → execução de testes.

Tipos de Testes

  • Manual: executado pelo analista.
  • Automatizado: scripts para validar.
  • Funcional: baseado em requisitos.
  • Não Funcional: desempenho, segurança, usabilidade.
  • Exploratório: sem roteiro fixo.
  • Regressão: garante que nada quebrou.
  • Smoke: checagem rápida do sistema.

Processos e Práticas

Ciclo de Vida do Bug: Novo → Em análise → Aberto → Em progresso → Resolvido → Reaberto → Fechado

Casos de Teste:

  • Identificador: CT-001
  • Pré-condições
  • Passos
  • Resultado esperado

BDD (Gherkin):

Dado que o usuário está na tela de login
Quando inserir usuário e senha válidos
Então deve acessar o sistema

Testes Não Funcionais

  • Performance: tempo de resposta.
  • Carga: múltiplos usuários.
  • Segurança: vulnerabilidades (SQLi, XSS).
  • Usabilidade: experiência do usuário.

Ferramentas Comuns

  • Automação: Selenium, Cypress, Playwright
  • APIs: Postman, RestAssured
  • Performance: JMeter, Gatling
  • Gestão: Jira, TestRail
  • CI/CD: Jenkins, GitHub Actions

Boas Práticas

  • Versionar scripts (Git).
  • Reportar bugs com clareza (passos + evidências).
  • Shift Left Testing (testar cedo).
  • Métricas de qualidade (cobertura, falhas críticas).

Glossário Rápido

  • Smoke Test: teste rápido de estabilidade.
  • Regression Test: garante que nada quebrou.
  • Exploratory Test: investigação livre.
  • CI/CD: Integração e Entrega Contínua.
  • Defeito/Bug: comportamento inesperado.
  • Hotfix: correção rápida em produção.

Glossário Rápido 2

  • Presença de Defeitos: testar mostra falhas, não garante ausência delas.
  • Testes Exaustivos: impossível testar tudo, priorize por risco.
  • Testes Antecipados: detectar cedo economiza tempo e custo.
  • Defeitos se Agrupam: maioria ocorre em poucos módulos.
  • Testes se Degradam: repetição reduz eficácia — atualize-os.
  • Contexto: cada projeto exige abordagem diferente.
  • Ausência de Defeitos ≠ Sucesso: sistema pode não atender o negócio.
  • Testes Funcionais: verificam o que o sistema faz (Unidade, Integração, Sistema, Aceite, Regressão, Smoke).
  • Testes Não Funcionais: verificam o quão bem (Performance, Carga, Stress, Usabilidade, Segurança, Compatibilidade).
  • Objetivos: detectar falhas, reduzir riscos, validar requisitos e gerar confiança.
  • Níveis de Teste: Componente (dev), Integração (módulos), Sistema (comportamento geral), Aceite (validação de negócio).
  • Processo de Teste: Planejar → Monitorar → Analisar → Modelar → Implementar → Executar → Concluir.
  • Caixa-Preta: baseada em requisitos (Partições, Valores-Limite, Tabela de Decisão, Estados).
  • Caixa-Branca: baseada no código (Cobertura de instruções e decisões).
  • Baseada em Experiência: Exploratório, Intuitivo, Defeitos Conhecidos.
  • BDD: define comportamento antes do código (Dado–Quando–Então).
  • TDD: teste vem antes do código – Red, Green, Refactor.
  • ATDD: testes de aceitação definidos com cliente antes do desenvolvimento.
  • Testador: pensa criticamente, revisa planos, prepara dados e configura ambiente.
  • Análise de Teste: o que testar.
  • Modelagem de Teste: como testar.
  • Teste Estático: sem executar código (revisões e análise estática); detecta falhas cedo.
  • Teste Dinâmico: executa o sistema para encontrar falhas.
  • Erro: engano humano.
  • Defeito: erro no código.
  • Falha: execução de um defeito.
  • Valor Limite: testa extremos (abaixo e acima dos limites).
  • Partição de Equivalência: divide entradas válidas e inválidas.
  • Estimativa 3 Pontos: (Otimista + 4×Provável + Pessimista) ÷ 6.
  • Produtos de Teste: Caso (resultados esperados), Roteiro (instruções), Carta (exploratórios), Conjunto (scripts executados).
  • Quadrantes Ágeis: Usabilidade e Funcional (negócio), Componentes e Confiabilidade (tecnologia).
  • Falhas Comuns: Componente (lógica), Integração (comunicação), Sistema (regras), Aceite (necessidades do usuário).
